テトリスは単純に見える — 実装するまでは。

実際のテトリスのループには、時間（tick）、並行する入力（キー操作）、状態遷移（衝突、固定、ライン消去）、そして非決定性（ピース生成）がある。多くの命令型の設計では、これらの関心事が共有された可変状態の中で絡み合い、次のようなバグを生みやすい:

- 再現が難しい（タイミング依存）
- テストが難しい（ロジックと副作用が混在）
- デバッグが難しい（リプレイが決定的でない）

**Monadris** は、これらの失敗モードを「もっと気をつける」問題ではなく、アーキテクチャの問題として扱う。中心となる考えは、*純粋な*コア（ドメイン＋ルール）と*不純な*アプリ層（I/O＋タイミング＋描画）を**物理的に分離**し、ゲームをイベント駆動の決定的な状態機械としてモデル化することである。

## 「安全」とはここで何を意味するか

「関数型設計がテトリスを安全にする」と言うとき、「FP が道徳的に優れている」という意味ではない。アーキテクチャによって、ある種のバグの混入が難しく（あるいは不可能に）なる、という意味である:

- 並行する入力／タイマーのロジックをまたいで共有される、可変なゲーム状態が存在しない
- 状態遷移の経路が単一で直列化されている（イベントを入れて、状態を出す）
- コアのルールは決定的で、ランタイム・時計・端末なしでテストできる
- 適切なイベントを記録すれば、リプレイを決定的にできる

## 命令型テトリスが実際に壊れがちな理由

tick と入力が共有された可変状態を奪い合うときの、よくある失敗モード:

- 時刻 *t* にプレイヤーが MoveDown を押す
- ほぼ同時に Tick が届く
- 両方のハンドラが同じ状態（落下中ピースの同じ位置）を読む
- 両方が「下に移動」を適用して書き戻す

スケジューリング次第で、次のようなことが起こりうる:

- 二重に落下する
- 古い位置で衝突判定をする
- 固定／消去が「早すぎる」タイミングで起こる
- デバッグすると消えるバグ（典型的なハイゼンバグ）

Monadris は「タイミングをうまく合わせる」ことを目指さない。**単一の順序を定義する**: イベントはキューに入れられ、ゲームループが一つずつ処理する。

## 1. コアを純粋な状態機械として固定する

設計の中心には厳格なルールがある:

> コアのゲームロジックは、(State, Input) から State への単なる関数である。

これはプロジェクトの第一級の特徴として明示されている。

実際には、コアは決して次のことをしない:

- 時計を読む
- 入力デバイスをポーリングする
- 端末に書き込む
- ファイルやデータベースとやり取りする

それらは別の場所で起こる。

この境界を固定すれば、コアは推論しやすくなる:

- 同じ状態 ＋ 同じ入力 → 同じ次の状態
- 「隠れた」状態の源（時間・乱数・I/O）がない

これが、信頼できるテストと信頼できるリプレイの両方の土台になる。

## 2. ADT で不正な状態と分岐の漏れを作りにくくする

ゲームループでは「入力」や「状態」は時間とともに膨らみがちだ（ポーズ、終了、ソフトドロップ、ハードドロップ、回転など）。それらを代数的データ型（Scala 3 の enum）として表現すると、パターンマッチが網羅的になるため効果が大きい。

アーキテクチャ上の狙いはシンプルだ:

- 新しい入力のバリアントを追加する → コンパイラが、対応すべき箇所へ導いてくれる

これはスタイルの話ではない。プロジェクトが進化する中で、変更の影響範囲を制御することの話である。

## 3. 純粋性の境界を、規約ではなくチームの制約として強制する

Monadris は「開発者が規律を守ってくれる」ことに頼らない。純粋なコアで「変更（mutation）禁止」の姿勢を強制することで、特定の近道を高くつくものにする:

- var なし
- null なし
- throw なし
- return なし

…そしてこれをコアモジュールで*コンパイル時に*行う。

重要なニュアンス: これは JVM 上のすべてが数学的に純粋であることを「証明」するわけではない。やっているのは、「ここだけ例外的に」や「手早く可変キャッシュをここに」といった形で**アーキテクチャの境界が崩れていくのを防ぐ**ことである — クリーンな設計はたいてい、そうやって時間とともに劣化していく。

## 4. 並行性は境界で、状態遷移は直列化する

入力と tick は本質的に並行である。設計上の選択はこうだ:

- 並行性は**イベントの生成**でのみ許す
- **状態遷移**はシングルスレッドで決定的に保つ

概念的には:

1. 入力ファイバーがイベントをキューに積む
2. タイマーファイバーが tick イベントを同じキューに積む
3. 一つのゲームループがイベントを消費し、update(state, input) を順番に適用する

この最後の点が「安全のレバー」である: 次の状態を生成する経路が、ちょうど一つだけ存在する。

## 5. 決定的なリプレイ: 入力の記録だけでは足りない

よくあるリプレイの間違いは、ユーザーの入力だけを記録することだ。ゲームに非決定性（例: 次のピース生成）が含まれた途端、それは破綻する。

決定的なリプレイが機能するのは、次のときである:

> 「入ってくるイベント」が、「出ていく状態」を再構築するのに十分であるとき。

大きく二つの方法がある:

- 乱数を決定的にする（シード付き RNG を状態に通す）、あるいは
- **非決定的な結果**（例: 実際に出現したピース）をイベントとして記録し、リプレイ時にそれを与え直す

## 6. テストが直接の状態遷移テストになる

コアが (State, Input) => State になると、テストは「統合テストっぽい」ものではなく、真にユニットレベルになる:

- 初期状態を作る
- 入力を適用する
- 新しい状態をアサートする

時計のモックも、端末のハーネスも、非同期の不安定さもない。

これは特に次の検証で価値がある:

- 衝突判定
- ライン消去
- スコア／レベル遷移
- ゲームオーバー条件
- リプレイの再構築

## ZIO を使う理由（簡潔に）

ZIO はこのアーキテクチャに自然に適合する。「不純なシェル」の関心事を、明示的かつ合成可能にするからだ:

- 構造化並行性のための **Fiber**（入力と tick を並行な生産者として）
- 安全でノンブロッキングなイベントバッファリングのための **Queue**
- **純粋な関数**と**副作用**の明確な分離
- コアを汚さずに、副作用の境界をテストできる支援

結果として、並行性は境界にとどまり、コアは決定的なまま保たれる。

## まとめ: 「FP が安全にする」のではなく、境界が安全にする

Monadris が「関数型」であるのは、本当に重要な意味でのアーキテクチャ的な関数型である:

- **純粋なコア**: 決定的なルールと変換
- **不純なシェル**: 入力・時間・描画・永続化のための ZIO 効果
- **イベントループ**: 並行な生産者、直列化された消費者
- **リプレイ**: 状態を決定的に再構築するのに必要なものを記録する
- **ガードレール**: コアを純粋に保ち続けるコンパイル時の制約

これらの境界こそが、システムを安全で、進化させやすくするものである。
