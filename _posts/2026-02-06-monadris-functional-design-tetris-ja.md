テトリスは単純に見える。実際に実装してみるまでは。

本物のテトリスのループには、時間（tick）、同時に飛んでくる入力（キー操作）、状態遷移（衝突・固定・ライン消去）、そして非決定性（ピースの生成）が詰まっている。これを命令型で素直に書くと、すべてが共有された可変状態の上で絡み合い、次のようなバグを生みやすい。

- 再現できない（タイミング依存）
- テストしづらい（ロジックと副作用が混ざる）
- デバッグできない（リプレイが決定的に再生されない）

**Monadris** は、こうした不具合を「気をつければ防げる」問題ではなく、設計（アーキテクチャ）の問題として捉える。狙いはシンプルだ。*純粋な*コア（ドメインとルール）と*不純な*アプリ層（I/O・タイミング・描画）を**物理的に分け**、ゲームをイベントで駆動する決定的な状態機械として組み立てる。

## ここで言う「安全」とは

「関数型だと安全になる」と言っても、FP が偉いという話ではない。設計の力で、ある種のバグを「混入しにくく」、あるいは「そもそも混入できなく」できる、という意味だ。

- 並行に動く入力やタイマーが、可変なゲーム状態を共有しない
- 状態を変える経路がただ一本に直列化されている（イベントを入れると、状態が出てくる）
- コアのルールは決定的で、ランタイムも時計も端末もなしにテストできる
- 必要なイベントさえ記録すれば、リプレイを決定的に再生できる

## 命令型のテトリスが現実に壊れる理由

tick と入力が、同じ可変状態を奪い合う。よくある壊れ方はこうだ。

- 時刻 t にプレイヤーが MoveDown を押す
- ほぼ同時に Tick が届く
- どちらのハンドラも同じ状態（落下中ピースの同じ座標）を読む
- どちらも「下に動かす」を適用して書き戻す

スケジューリング次第で、こんなことが起きる。

- 一度に二段落ちる
- 古い座標のまま衝突判定してしまう
- 固定やライン消去が「早すぎる」タイミングで走る
- デバッグすると消えてしまうバグ（典型的なハイゼンバグ）

Monadris は「タイミングをうまく合わせよう」とはしない。代わりに**順序をひとつに決める**。イベントはいったんキューに積み、ゲームループが一つずつ取り出して処理する。

## 1. コアを純粋な状態機械として固める

設計の真ん中には、ひとつの厳しいルールがある。

> コアのゲームロジックは、(State, Input) を受け取って State を返すだけの関数である。

これはプロジェクトの中核的な特徴として、はっきり打ち出しているものだ。

つまりコアは、次のことを一切しない。

- 時計を読む
- 入力デバイスを覗く
- 端末に書き出す
- ファイルや DB を触る

そうした仕事は、すべて外側でやる。

この境界さえ固めてしまえば、コアは一気に追いやすくなる。

- 同じ状態に同じ入力を与えれば、必ず同じ次の状態になる
- 時間・乱数・I/O といった「隠れた入力」が存在しない

これが、テストの信頼性とリプレイの信頼性、その両方の土台になる。

## 2. ADT で「ありえない状態」と「書き忘れた分岐」を防ぐ

ゲームループの「入力」や「状態」は、開発が進むほど増えていく（ポーズ、終了、ソフトドロップ、ハードドロップ、回転……）。これらを代数的データ型（Scala 3 の enum）で表すと、パターンマッチが網羅的になって効いてくる。

狙いは単純だ。

- 入力の種類をひとつ増やすと、コンパイラが「ここも対応して」と漏れなく指摘してくれる

見た目の好みの話ではない。プロジェクトが育つほど大きくなる「変更の波及範囲」を、抑え込むための仕組みだ。

## 3. 純粋さの境界は、規約ではなく「制約」で守る

Monadris は「開発者が気をつけてくれるはず」には頼らない。純粋なコアでは「書き換え禁止」を強制し、横着な近道を割に合わなくしている。

- var 禁止
- null 禁止
- throw 禁止
- return 禁止

しかもこれを、コアモジュールでは*コンパイル時*に効かせる。

ひとつ補足しておくと、これで JVM 上のすべてが数学的に純粋だと「証明」できるわけではない。できるのは、「ここだけは例外で」「とりあえず可変キャッシュを置いて」といった綻びから**設計の境界を守る**ことだ。きれいな設計は、たいていそういう小さな例外から崩れていく。

## 4. 並行処理は外側に、状態遷移は直列に

入力と tick は、そもそも並行に起きる。そこでこう切り分ける。

- 並行に動かしてよいのは**イベントを生み出すところ**だけ
- **状態を遷移させるところ**は、シングルスレッドで決定的に保つ

イメージはこうだ。

1. 入力用のファイバーが、イベントをキューに積む
2. タイマー用のファイバーが、tick イベントを同じキューに積む
3. ゲームループはひとつだけで、イベントを順に取り出して update(state, input) を適用する

この最後の一点が安全性の要になる。次の状態を生み出す経路が、きっかり一本しかない。

## 5. 決定的なリプレイ：入力を記録するだけでは足りない

リプレイでありがちな失敗が、ユーザーの入力だけを記録してしまうことだ。ゲームに非決定性（たとえば次のピースの生成）が入った瞬間、これは破綻する。

リプレイが決定的に成立するのは、次の条件を満たすときだけだ。

> 「入ってくるイベント」だけで、「出ていく状態」を再構築できること。

やり方は大きく二つある。

- 乱数のほうを決定的にする（シード付き RNG を状態に通す）
- あるいは、**非決定的に決まった結果**（実際に出現したピースなど）をイベントとして記録しておき、リプレイ時にそれを与え直す

## 6. テストが、そのまま状態遷移のテストになる

コアが (State, Input) => State になると、テストは「半分統合テスト」のようなものから、純粋なユニットテストに変わる。

- 初期状態を用意する
- 入力をひとつ適用する
- 出てきた状態を検証する

時計のモックも、端末のハーネスも、非同期ゆえの不安定さもいらない。

特にこのあたりの検証で効いてくる。

- 衝突判定
- ライン消去
- スコアやレベルの遷移
- ゲームオーバー判定
- リプレイの再構築

## なぜ ZIO なのか（手短に）

ZIO はこの設計と相性がいい。「不純なシェル」の関心事を、明示的で組み合わせやすい形にしてくれるからだ。

- 構造化並行性のための **Fiber**（入力と tick を、並行な生産者として走らせる）
- 安全でノンブロッキングなイベントバッファとしての **Queue**
- **純粋な関数**と**副作用**をはっきり分けられる書き味
- コアを汚さずに、副作用の境界だけをテストできる仕組み

おかげで、並行性は外側にとどまり、コアは決定的なまま保たれる。

## まとめ：安全にしているのは「FP」ではなく「境界」

Monadris が「関数型」なのは、いちばん効くところでアーキテクチャが関数型になっている、という意味だ。

- **純粋なコア**：決定的なルールと変換
- **不純なシェル**：入力・時間・描画・永続化を担う ZIO 効果
- **イベントループ**：生産者は並行、消費者は直列
- **リプレイ**：状態を決定的に復元するのに必要なものだけを記録する
- **ガードレール**：コアの純粋さを保ち続けるコンパイル時の制約

安全で、育てやすい。それを支えているのは、この「境界」のほうだ。
